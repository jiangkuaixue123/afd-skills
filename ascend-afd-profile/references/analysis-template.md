# 分析输出模板

写 AFD profile 结论时，优先用这个模板，保持输出稳定、可对比。

## 实验扫描结果

先说明：

- 一共发现多少组实验
- 哪些实验目录完整
- 哪些实验缺少 `run_params.txt`、`profile/attention` 或 `profile/ffn`

## 单实验结论

对每组实验至少写四项：

- `配置摘要：` 从 `run_params.txt` 的 `Test Scenario:` 段落中提取出的关键 `key=value` 字段
- `Attention 侧观察：` `A2e`、`E2a`、Attention 相关算子的时延
- `FFN 侧观察：` `A2e`、`E2a`、`GroupMatmul`、`MoeDispatch`、`MoeCombine` 的时延
- `综合判断：` 这组实验更像是哪类瓶颈

如果存在解析不确定性，补一行：

- `解析说明：` 哪些字段是从 `Test Scenario:` 明确提取的，哪些内容只是背景文本、未纳入配置对比

如果需要强调关键经验规则，再补一行：

- `瓶颈提示：` 若某一侧 `A2e` 或 `E2a` 明显更长，优先判断另一侧是当前时延瓶颈

## 跨实验对比

优先回答这些问题：

- 哪些配置字段变化后，性能现象发生了明显变化
- 哪些实验的 Attention/FFN 更平衡
- 哪些实验更像 handoff 问题而不是单侧计算问题
- 是否存在“配置变化了，但 profile 现象没变”的情况
- 哪些实验中 `A2e` / `E2a` 的长尾从一侧转移到了另一侧

## 根因候选

最多列三个：

1. 最可能落在关键路径上的直接原因
2. 更上层的结构性原因，例如 handoff 设计、调度方式、batch 划分
3. 次要因素或当前还无法确认的因素

## 下一步建议

建议一次只改一个变量，便于回归因果：

1. 补齐 stage marker，显式标记 Attention、handoff、FFN
2. 选两组配置最接近、现象差异最大的实验做重点对比
3. 调整一个 batching / routing / parallelism 参数后重新 profile
4. 对慢实验和快实验做同维度并排比较

## 建议的表述方式

每条证据尽量写成下面的样子：

- `观察：` profile 中实际看到的事实
- `解释：` 这个事实最可能说明什么
- `置信度：` 高 / 中 / 低

对于 `A2e` / `E2a`，优先写成：

- `观察：` Attention 或 FFN 侧哪一个 `A2e` / `E2a` 更长
- `解释：` 按当前经验规则，这通常说明对侧是当前时延瓶颈
- `置信度：` 高 / 中 / 低

## 结尾句建议

最后给一句明确的下一步判断，例如：

“当前证据更支持 handoff 串行化，而不是 FFN 纯计算过慢；下一步最值得做的是在相同 batch 配置下，对两组实验做带 stage marker 的 Attention/FFN 时间线对比。”
